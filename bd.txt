## 1. Informacion General

**Proyecto:** Lotifyx - Plataforma de Marketplace Multifuncional

**Cliente:** Lotifyx S.A.C.

**Tecnologia de Base de Datos:** PostgreSQL 16+

**Motor de Base de Datos:** PostgreSQL

**Version del Documento:** 1.0

**Fecha de Elaboracion:** 10 de junio de 2026

**Equipo Responsable:** Departamento de Arquitectura de Software

**Descripcion del Proyecto:** Lotifyx es una plataforma de marketplace integral que permite la compra y venta de productos, subastas en tiempo real, ventas por lote, cotizaciones personalizadas, servicios, y gestion de pagos con proteccion de fondos mediante sistema escrow. La plataforma opera en el mercado peruano, utilizando codigos de verificacion RENIEC y comprobantes electronicos.

**Alcance del Documento:** Este documento describe el diseno completo de la base de datos relacional del sistema Lotifyx, abarcando el modelo conceptual, logico y fisico, diccionario de datos, reglas de integridad, indices, estrategias de seguridad, respaldo y escalabilidad. El diseno contempla 67 tablas con soporte para todas las funcionalidades del MVP (Minimum Viable Product) y proyecciones para fases futuras.

---

## 2. Objetivo del Documento

El presente documento tiene como objetivos fundamentales:

1. **Definir la arquitectura de datos** que soporte todas las operaciones del marketplace Lotifyx, incluyendo comercio electronico, subastas, ventas por lote, cotizaciones y servicios.

2. **Establecer el modelo de datos normalizado** que garantice la integridad referencial, evite la redundancia y permita consultas eficientes en un entorno de alta concurrencia.

3. **Documentar exhaustivamente cada entidad** con sus campos, tipos de datos, restricciones y relaciones, sirviendo como referencia unica para los equipos de desarrollo backend, frontend y QA.

4. **Proveer la base para la generacion automatica** de migraciones, modelos ORM y documentacion de API a partir de la definicion del esquema.

5. **Facilitar la trazabilidad** entre los requerimientos funcionales del sistema y las entidades de base de datos que los implementan.

6. **Garantizar el cumplimiento normativo** mediante la documentacion de estrategias de auditoria, cifrado de datos sensibles y politicas de retencion.

7. **Servir como insumo para auditorias** de seguridad, revisiones de arquitectura y procesos de onboarding de nuevos desarrolladores al proyecto.

## 5. Modelo Conceptual

### 5.1 Entidades Nucleares del Negocio

El modelo conceptual de Lotifyx se fundamenta en 11 entidades nucleares que representan los conceptos centrales del negocio:

1. **Usuario (User):** Entidad central que representa a cualquier persona registrada en la plataforma. Un usuario puede actuar como comprador, vendedor o ambos. Contiene credenciales de acceso, datos de contacto y estado de verificacion.

2. **Producto (Product):** Representa un bien tangible o intangible publicado por un vendedor. Puede ser de tipo venta directa, subasta o lote. Incluye precio, descripcion, imagenes, variantes y condiciones de venta.

3. **Pedido (Order):** Transaccion de compra que agrupa uno o mas productos de diferentes vendedores. Contiene el estado de pago, direccion de envio y costos asociados.

4. **Subasta (Auction):** Modalidad de venta donde los compradores compiten mediante pujas durante un periodo definido. Requiere un precio base, incremento minimo y garantia de participacion.

5. **Venta por Lote (Lot Sale):** Modalidad de compra colectiva donde el precio final depende de la cantidad de participantes. Cada participante se compromete a comprar una cantidad especifica.

6. **Cotizacion (Quote):** Solicitud de presupuesto personalizado donde compradores describen sus necesidades y vendedores envian propuestas con precios y condiciones.

7. **Fondos (Funds):** Billetera virtual del vendedor que registra el saldo disponible, pendiente y en disputa. Centraliza todas las transacciones financieras de la plataforma.

8. **Disputa (Dispute):** Proceso de resolucion de conflictos entre comprador y vendedor, con intervencion administrativa, mensajes y evidencias documentales.

9. **Escrow (Escrow):** Mecanismo de proteccion que retiene el pago del comprador hasta que se confirme la recepcion satisfactoria del producto.

10. **Chat (Chat):** Sistema de mensajeria directa entre dos participantes, vinculado opcionalmente a un pedido, que permite la comunicacion en tiempo real.

11. **Administracion (Admin):** Conjunto de entidades que gestionan roles administrativos, permisos, configuraciones del sistema, sanciones y auditoria.

### 5.3 Convenciones del Diagrama

- **Flecha simple (--->):** Relacion uno a muchos (1:N). La entidad del lado de la flecha tiene la clave foranea.
- **Linea doble (<==>):** Relacion uno a uno (1:1).
- Las entidades en la misma columna vertical estan agrupadas por dominio funcional.
- Las relaciones opcionales se indican con "(opcional)".

| 1 | roles | 4 | id | - | users 1:N roles | Roles de usuario del sistema (comprador, vendedor, administrador). Define el nivel de acceso base de cada usuario registrado. |

### 7.5 Generacion de Claves Primarias

Todas las claves primarias del sistema se generan utilizando la funcion uuid_generate_v4() de la extension uuid-ossp. Esta estrategia proporciona las siguientes ventajas:

- **Independencia de secuencia centralizada:** Permite generar IDs desde multiples servidores de aplicacion sin coordinacion.
- **Escalabilidad horizontal:** Compatible con sharding y particionamiento futuro.
- **Seguridad:** Los UUID v4 son aleatorios, no predecibles secuencialmente.
- **Portabilidad:** Facilita la migracion de datos entre entornos.

## 11. Reglas de Integridad

### 11.1 Integridad Referencial

Todas las claves foraneas del sistema utilizan la politica ON DELETE RESTRICT por defecto para prevenir la eliminacion accidental de registros referenciados. Las excepciones documentadas son:

| Tabla Hija | Columna FK | Tabla Padre | Politica ON DELETE | Justificacion |
|------------|------------|-------------|--------------------|---------------|
| user_profiles | user_id | users | CASCADE | El perfil debe eliminarse si el usuario es eliminado |
| seller_profiles | user_id | users | CASCADE | El perfil de vendedor se elimina con el usuario |
| social_accounts | user_id | users | CASCADE | Cuentas sociales se eliminan con el usuario |
| product_images | product_id | products | CASCADE | Imagenes se eliminan con el producto |
| product_variants | product_id | products | CASCADE | Variantes se eliminan con el producto |
| product_sale_conditions | product_id | products | CASCADE | Condiciones se eliminan con el producto |
| cart_items | cart_id | carts | CASCADE | Items se eliminan con el carrito |
| order_items | order_id | orders | CASCADE | Items se eliminan con la orden |
| plan_features | plan_id | plans | CASCADE | Caracteristicas se eliminan con el plan |
| service_images | service_id | services | CASCADE | Imagenes se eliminan con el servicio |
| auction_bids | auction_id | auctions | CASCADE | Pujas se eliminan con la subasta |
| dispute_messages | dispute_id | disputes | CASCADE | Mensajes se eliminan con la disputa |
| chat_messages | conversation_id | chat_conversations | CASCADE | Mensajes se eliminan con la conversacion |
| notifications | user_id | users | CASCADE | Notificaciones se eliminan con el usuario |
| fund_transactions | user_id | users | CASCADE | Transacciones se eliminan con el usuario |

En el resto de relaciones se utiliza RESTRICT para evitar eliminaciones accidentales de datos de negocio criticos como pedidos, pagos, disputas y registros de auditoria.

### 11.4 Restricciones NOT NULL en Campos Criticos

Todos los campos de tipo FOREIGN KEY son NOT NULL excepto en los siguientes casos donde se permite NULL para soportar relaciones opcionales:

- categories.parent_id: La categoria raiz no tiene padre
- products.approved_by: Producto puede no estar aprobado aun
- orders.shipping_address: Productos digitales no requieren direccion
- payment_proofs.bank_account_id: Pago puede no estar asociado a cuenta
- escrow_records.released_at: Solo se completa al liberar fondos
- auctions.winner_id: Solo se completa al finalizar la subasta
- disputes.resolved_at: Solo se completa al resolver la disputa

### 11.5 Valores por Defecto

Todos los valores por defecto documentados en el diccionario de datos (Seccion 9) se implementan a nivel de base de datos mediante la clausula DEFAULT en la definicion de columna. Los valores monetarios, contadores y acumuladores siempre se inicializan en 0. Los estados de flujo se inicializan en su valor inicial del ciclo de vida correspondiente.

### 12.2 Estrategia de Optimizacion

- **Indices compuestos:** Se utilizan donde las consultas frecuentemente filtran por multiples columnas (ej: notifications(user_id, is_read), chat_messages(conversation_id, is_read)).
- **Indices en claves foraneas:** Todas las FK tienen indices para optimizar JOINs.
- **Indices en columnas de estado:** Permiten consultas de dashboard y administracion.
- **Indices en columnas de fecha:** Facilitan consultas de rango y reportes periodicos.
- **VACUUM y ANALYZE:** Se programaran tareas de mantenimiento automatico para actualizar estadisticas del planificador.

### 12.3 Cobertura de Indices

El sistema cuenta con 63 indices que cubren:
- 100% de claves foraneas
- 100% de columnas utilizadas en filtros WHERE frecuentes
- 100% de restricciones UNIQUE
- Columnas de ordenamiento (sort_order, created_at)
- Columnas de soft delete (deleted_at)

### 14.1 Roles de Base de Datos

### 14.2 Cifrado de Datos

| Dato Protegido | Algoritmo | Implementacion | Justificacion |
|----------------|-----------|----------------|---------------|
| password_hash (users) | bcrypt | pgcrypto crypt() con gen_salt('bf', 10) | Proteccion de credenciales de usuario |

### 14.3 Row-Level Security (RLS)

No se implementa Row-Level Security en la fase MVP. La autorizacion a nivel de fila se maneja en la capa de aplicacion mediante el sistema RBAC. Se evalua su implementacion en fase 2 para:

- Aislamiento de datos entre vendedores
- Restriccion de acceso a datos personales
- Segmentacion de datos por region

### 14.4 Auditoria

La tabla audit_logs proporciona un registro inmutable de todas las operaciones criticas del sistema:

- CREATE: Registra la entidad creada con sus valores iniciales (new_values)
- UPDATE: Registra valores anteriores (old_values) y nuevos (new_values)
- DELETE: Registra valores anteriores (old_values) antes de la eliminacion
- LOGIN: Registra inicios de sesion con IP y User-Agent
- ACCIONES ADMINISTRATIVAS: Registra aprobaciones, rechazos, sanciones y revisiones

### 14.5 Proteccion contra Inyeccion SQL

- Todas las consultas utilizan parametros bind (prepared statements) desde la aplicacion
- No se construyen consultas SQL mediante concatenacion de strings
- El ORM de la aplicacion implementa escaping automatico de parametros

## 15. Estrategia de Respaldo y Recuperacion

### 15.1 Politica de Backups

| Tipo de Backup | Frecuencia | Destino | Retencion |
|----------------|------------|---------|-----------|
| Backup completo (pg_dump) | Diario (02:00 America/Lima) | Cloudflare R2 | 30 dias |
| Backup de schema unicamente | Semanal (lunes) | Cloudflare R2 | 90 dias |
| Backup pre-deploy | Antes de cada migracion de esquema | Cloudflare R2 | 60 dias |

### 15.2 Comando de Backup

```bash
pg_dump -h $DB_HOST -U $DB_USER -d lotifyx \
  -F c -b -v -f lotifyx_$(date +%Y%m%d_%H%M%S).dump
```

### 15.3 Procedimiento de Restauracion

```bash
# 1. Verificar que el entorno de destino esta limpio
psql -h $DB_HOST -U $DB_ADMIN -d postgres -c "DROP DATABASE IF EXISTS lotifyx;"
psql -h $DB_HOST -U $DB_ADMIN -d postgres -c "CREATE DATABASE lotifyx;"

# 2. Restaurar backup completo
pg_restore -h $DB_HOST -U $DB_ADMIN -d lotifyx \
  -v --no-owner --no-privileges lotifyx_YYYYMMDD_HHMMSS.dump

# 3. Verificar integridad
psql -h $DB_HOST -U $DB_ADMIN -d lotifyx -c "
  SELECT schemaname, relname, n_live_tup
  FROM pg_stat_user_tables
  ORDER BY n_live_tup DESC;
"
```

### 15.4 Point-in-Time Recovery (PITR)

No se implementa PITR en la fase MVP. Se requiere configurar WAL archiving y replicacion de streaming para habilitar esta capacidad en fase 2.

### 15.5 Plan de Recuperacion ante Desastres

| Escenario | RPO (Recovery Point Objective) | RTO (Recovery Time Objective) | Procedimiento |
|-----------|-------------------------------|------------------------------|---------------|
| Falla de servidor unico | < 24 horas | < 4 horas | Restauracion desde backup mas reciente en nueva instancia |
| Corrupcion de datos | < 24 horas | < 4 horas | Restauracion desde backup y verificacion de integridad |
| Perdida total de infraestructura | < 24 horas | < 8 horas | Provisionar nueva infraestructura y restaurar backup |

## 16. Reglas de Nomenclatura

### 16.1 Nombres de Tablas

- **Formato:** snake_case, plural en ingles
- **Ejemplos:** users, products, order_items, chat_conversations
- **Tablas de cruce:** {entidad_a}_{entidad_b} (ej: seller_badges, admin_role_permissions)
- **Tablas de configuracion:** nombre descriptivo en plural (ej: site_settings, smtp_configs)

### 16.2 Nombres de Columnas

- **Formato:** snake_case, descriptivo
- **Clave primaria:** id (en todas las tablas)
- **Clave foranea:** {tabla_referenciada_singular}_id (ej: user_id, category_id, product_id)
- **Fechas de auditoria:** created_at, updated_at
- **Fecha de borrado logico:** deleted_at
- **Fechas de accion:** {verbo}_{preposicion}_at (ej: approved_at, verified_at, released_at, resolved_at)
- **Banderas booleanas:** is_{adjetivo} (ej: is_verified, is_active, is_read, is_primary)
- **Contadores:** {sustantivo}_count (ej: views_count, sales_count)

### 16.3 Nombres de Restricciones

| Tipo | Patron | Ejemplo |
|------|--------|---------|
| Primary Key | pk_{tabla} | pk_users |
| Foreign Key | fk_{tabla}_{columna} | fk_products_user_id |
| Unique | uk_{tabla}_{columna(s)} | uk_users_email |
| Check | ck_{tabla}_{columna} | ck_reviews_rating |
| Index | idx_{tabla}_{columnas} | idx_products_slug |

### 16.4 Estandares Generales

- **Idioma:** Todos los nombres de tablas, columnas y restricciones en ingles
- **Separador:** Guion bajo (snake_case) exclusivamente, sin camelCase ni guiones
- **Sin prefijos:** No se utilizan prefijos como tbl_, v_, sp_ en tablas
- **Sin abreviaturas:** Nombres completos y descriptivos (quantity, no qty)
- **Consistencia:** El mismo concepto usa el mismo nombre en todas las tablas (created_at en las 67 tablas)

## 17. Consideraciones de Escalabilidad

### 17.1 Estrategia de Claves UUID v4

La adopcion de UUID v4 como clave primaria en todas las tablas proporciona las siguientes ventajas para la escalabilidad:

- **Escalabilidad horizontal:** Permite sharding sin conflictos de secuencias
- **Generacion descentralizada:** Multiples servidores de aplicacion generan IDs sin coordinacion
- **Independencia del servidor:** No depende de secuencias del motor de base de datos
- **Migracion simplificada:** Facilita replicacion y sincronizacion entre entornos

### 17.2 Particionamiento (Futuro)

Se preve implementar particionamiento por rango de fecha en las siguientes tablas:

| Tabla | Criterio de Particion | Prioridad |
|-------|----------------------|-----------|
| audit_logs | created_at (mensual) | Alta |
| notifications | created_at (mensual) | Alta |
| chat_messages | created_at (mensual) | Media |
| fund_transactions | created_at (trimestral) | Media |
| product_views | viewed_at (mensual) | Baja |

### 17.3 Sharding

No se implementa sharding en la fase MVP. Se evalua para fase 3 basado en los siguientes criterios:

- Volumen de datos supera capacidad de una unica instancia
- Latencia de consultas excede SLAs establecidos
- Necesidad de aislamiento geografico de datos

### 17.4 Read Replicas

Se contempla la implementacion de replicas de lectura en fase 2 para:

- Separar carga de consultas analiticas (reportes, dashboards)
- Mejorar rendimiento de busquedas de catalogo
- Distribuir carga de consultas en horas pico

### 17.5 ElasticSearch (Futuro)

Para la fase 2 se evalua la integracion con ElasticSearch para:

- Busqueda de texto completo en productos (titulo, descripcion, marca)
- Busqueda con filtros facetados (categoria, precio, condicion)
- Autocompletado de busquedas
- Relevancia y ranking de resultados

### 17.6 Estrategia de Conexiones

| Componente | Pool Size Recomendado | Timeout |
|------------|----------------------|---------|
| Aplicacion Backend | 20-50 conexiones por instancia | Statement: 30s |

### 18.10 Reemplazo de Tipos ENUM

El sistema no utiliza tipos ENUM nativos de PostgreSQL. Todas las enumeraciones se implementan mediante:

1. **Columnas VARCHAR** con restricciones CHECK a nivel de base de datos (ver Seccion 7.4)
2. **Validacion a nivel de aplicacion** para mensajes de error amigables
3. **Documentacion de valores permitidos** en el diccionario de datos (Seccion 9)

Esta decision de diseno se fundamenta en:

- Mayor flexibilidad para agregar valores sin ALTER TABLE
- Compatibilidad con herramientas de migracion y ORMs
- Evitar el acoplamiento a tipos propietarios de PostgreSQL
- Facilitar la migracion futura a otros motores de base de datos si fuera necesario
